## Контекст теми
Команда поділила 13 таблиць та роботу з кожною з них зі схеми на три функціональні блоки, де кожен учасник пише повний DML для свого набору таблиць:  щонайменше 10 INSERT INTO для кожної таблиці, 1 UPDATE and 1 DELETE (по можливості) до свого набору таблиць. При цьому враховані всі CONSTRAINT обмеження до значень, обмеження на видалення та зв'язки між таблицями, правила NOT NULL чи визначення щодо допустимих значень або навпаки визначене DEFAULT VALUE, зазначались різні реалістичні приклади для варіативності випадків. Також додавались коментарі до кожної секції та операції для пояснень, уточнюивались враховані обмеження чи можливі побажання до виправлення у попередньому ddl.sql для перспектив розвитку проекту. 
Потім всі три частини збираються у фінальний dml.sql з правильним порядком INSERT-ів. Контроль відповідності оригінальній схемі — звірка кожної таблиці з попередньо створеним ddl.sql (типи колонок, взаємозв'зяки, допустимі та дефолтні значення, обмеження). Скрипт був успішно виконаний у PostgreSQL.

## Коротке обґрунтування командного підходу
1. Як ви розподілили таблиці/сценарії наповнення між учасниками: 
За функціональними блоками, а а також з врахуванням зміни учасників команди для розподілу задач в команді.
- Блок 1 — фінансова частина (members + membership_plans + member_subscriptions).
- Блок 2 — операційне ядро (classes + attendance + equipment + junction + specializations).
- Блок 3 — індивідуальна робота з клієнтом та дані тренерів (personal training + fitness_goals + progress + trainers + junction with specializations).

Кожен блок — самодостатній набір повʼязаних таблиць та роботи з ними, тож учасник пише і INSERT INTO для кожної з своїх таблиць, і DELETE + UPDATE до них з коментарями чи зауваження щодо існуючим обмежень та правил, без потреби координуватися з іншими по дрібницях.
Ми обрали такий підхід для мінімізації залежностей між учасниками. Розподіл таблиць дуже схожий до попередньої задачі, тому кожному легше працювати з вже знайомими таблицями, окрім того таблиці поділені на блоки згідно їх зв'язку між собою, щоб одна людина враховувала найменше зв'язків одночасно і могла зосередитись суто на своїх задачах - це дозволило паралелізувати роботу, що було дуже ефективно та актуально через зміни у складі команди.

2. Чому вибрані саме такі тестові дані: 
Тестові дані обирались шляхом максимальної реалізації та охоплення всіх можливих випадків внесення даних, при цьому враховуючи всі правила та обмеження до значень у відповідних таблицям, а також взаємозв'язки між ними. Таблиці навмисно заповнювались в випадковому порядку рядків (тобто, member_id/goad_id не йшли по порядку), заповнювались різними комбінаціями та варіантами опису (наприклад для даних, що мають DEFAULT VALUE обиралось як написання вручну, так і через NULL / DEFAULT / пропуск), та перевірялась на реалістничність їх зв'язку згідно з взаємозв'язками таблиць.

4. Як перевіряли коректність і реалістичність DML-скриптів: 
- По-перше, dml.sql будувалось на основі вже раніше створеної ddl.sql — кожнен запис у DML звіряється щодо відповідності до DDL та взаємозв'язів у ній без редагування останньої, але з внесенням коментарів як нотаток з ідеями можливих покращень для розвитку проекту 
- По-друге, еревірявся формат для кожного значення та у випадку заявленого списку можливих значень - відповідність їм. Так само щодо обмежень NOT NULL чи дефолтних значень, а також усіх CONTRAINT та CHECK у DDL.
- По-третє, виконання скрипта у PostgreSQL-базі — якщо скрипт виконується без помилок і всі блоки  проходять, значить порядок заповнення таблиць та оновлення чи видалення даних в них виконано коректно.
