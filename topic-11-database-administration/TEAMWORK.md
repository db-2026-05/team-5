## Контекст теми
Створено чотири ролі: manager_role, trainer_role, receptionist_role, member_role. Для кожної ролі створено реальних користувачів — імена тренерів і клієнтів взяті з бази даних, менеджер та адміністратори мають вигадані імена. Права налаштовано для всіх 13 таблиць з урахуванням функції кожної ролі. Додатково реалізовано два REVOKE-запити — для демонстрації відкликання прав після перегляду логіки доступу. Наприкінці скрипту — опційний блок очищення.

## Коротке обґрунтування командного підходу

1. Чому обрали саме такі ролі та рівні доступу:
Чотири ролі відображають реальну структуру фітнес-центру. Менеджер має повний доступ як відповідальний за цілісність системи. Тренер читає більшість таблиць і може оновлювати статус обладнання — це відповідає його операційним задачам. Адміністратор керує реєстраціями, розкладом та бронюванням, але не може видаляти записи. Клієнт має лише читання релевантних таблиць — без доступу до даних інших клієнтів. Таблиці з фінансовими даними закриті для тренерів, таблиці з інформацією про фізичний стан клієнтів — для адміністраторів.

2. Як розподілили відповідальність за безпекову модель:
Завдання виконувалось однією людиною, тому вся модель побудована як єдина узгоджена система. Логіка доступу для кожної таблиці обґрунтовується через коментарі безпосередньо у скрипті. Два REVOKE-запити демонструють перегляд прав після первинного налаштування — адміністратору відкликано право оновлювати спеціалізації та створювати нові підписки.

3. Як перевірили, що ролі реально відрізняються правами:
Скрипт виконувався у PostgreSQL — всі запити пройшли без помилок. Коректність прав звірялась з логікою предметної області: кожна роль отримала рівно ті операції, які відповідають її функції в системі. REVOKE-запити перевірялись окремо — система повернула успішний результат виконання.
