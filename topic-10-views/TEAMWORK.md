## Контекст теми
Команда поділила 13 таблиць на два функціональні блоки. Кожен учасник реалізував повний набір з 8 типів views для свого блоку таблиць: horizontal, vertical, mixed, join-based, subquery-based, UNION-based, view-from-view та WITH CHECK OPTION. До кожного view додано demo-SELECT для перевірки результату. Валерія відповідала за клієнтський та фінансовий блок. Анжела відповідала за два блоки — заняття з обладнанням та тренерів з індивідуальними тренуваннями. Всі views зібрано у фінальний views.sql у порядку: спочатку блок Валерії, потім два блоки Анжели.

## Коротке обґрунтування командного підходу

1. Як ви розподілили типи views між учасниками:
Розподіл відбувся за тими самими функціональними блоками, що й у попередніх завданнях — кожен учасник працював зі знайомими таблицями. Кожен реалізував усі 8 типів views для свого набору таблиць, що дозволило паралелізувати роботу без необхідності координуватися між блоками.

2. Чому ці views важливі для предметної області:
Views відображають реальні потреби фітнес-центру. Горизонтальні та вертикальні views обмежують доступ до даних залежно від ролі — наприклад, публічний розклад занять чи лише активні підписки. Join-views дають зведену картину по клієнту або тренеру без повторення логіки запиту. Subquery-views виділяють конкретні сегменти — преміум-клієнти, добре оснащені заняття. UNION-views збирають різнорідні дані в одну вибірку — наприклад, протерміновані та скасовані підписки або клієнти з покинутими цілями та недоступні тренери. Views з CHECK OPTION захищають критичні поля від некоректних змін — статус обладнання чи рівень тренера.

3. Як перевіряли практичну цінність і коректність кожного view:
Кожен view перевірявся через demo-SELECT одразу після створення. Коректність звірялась із ddl.sql — типи колонок, взаємозв'язки між таблицями та допустимі значення. View-from-view додатково перевірявся на залежність від базового view. Фінальний views.sql був успішно виконаний у PostgreSQL без помилок.
